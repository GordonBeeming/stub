01 why build this

Most short-link and burn-note services run in someone else's tenant, behind someone else's account model. If you already have a domain and a Cloudflare account, this is a weekend of work and $0/month forever.

// Existing tools

Hosted services want an account you didn't ask for. Self-hosted alternatives tend to need a VPS. Cloudflare redirect rules hard-code destinations with no UI, no stats, no notes.

// stub

Deploy to your account, point your domain at it, passkey-log-in as the only user. Create short links. Burn notes. Done. Anyone can fork and stand up their own in ~10 minutes.

03 one app, or two?

One app. Ship them together.

Both features are the same primitive: a short random key that resolves to a stored payload, optionally with TTL and single-use semantics. A short link is just a note whose payload is { type: "redirect", url }. A burn note is { type: "note", ciphertext, burnOnRead: true }.

Same D1 schema, same dashboard, same passkey login, same Worker, same deploy. Splitting them doubles your maintenance burden for zero architectural gain. The single-tenant constraint makes it even cleaner — no tenant isolation to worry about.

04 architecture

Single Worker binds every resource. Static assets ship in the same bundle. No Pages project, no extra deploys.

07 auth flow — single tenant, passkey-first

Configured with a single environment variable: OWNER_EMAIL. Only that address can ever sign in.

01 / bootstrap
Request magic link
Owner enters email. Server checks it matches OWNER_EMAIL, mints one-time token to KV (10 min TTL).
02 / verify
Click link in email
Token validated, consumed, session issued. Signed JWT in HttpOnly cookie.
03 / enroll
Register passkey
First login prompts WebAuthn enrollment — Touch ID, Windows Hello, YubiKey. Stored in D1.
04 / return
Passkey sign-in
Subsequent logins skip email. Passkey only. Magic link kept as recovery fallback.

08 feature surface

Short links

Custom slug or auto-generated nanoid. Optional expiry, max-click limit, click analytics (hashed IP, UA family, referrer).

Secret notes

Plaintext encrypted client-side with WebCrypto (AES-GCM), key in URL fragment. Server sees only ciphertext.

Burn on read

Row deleted on first successful decrypt. TTL-based expiry as a backstop. Cron prunes the floor weekly.

QR codes

Any link or note URL can be rendered as a QR inline. Useful for sharing in person without typing.

Owner dashboard

Create, list, revoke. Realtime click/view stats. Bulk delete. Export to JSONL for backup.

Public create

Optional mode: anyone can create a burn-note (no login, Turnstile protected). Short links stay owner-only.

09 standing your own up

Whole-repo forker experience. ~10 minutes from clone to live domain.

# clone & configure $ gh repo fork GordonBeeming/stub --clone $ cd stub && pnpm install $ cp wrangler.example.toml wrangler.toml # set your single owner email $ npx wrangler secret put OWNER_EMAIL > you@yourdomain.com # provision resources (one command per) $ npx wrangler d1 create stub-db $ npx wrangler kv namespace create SESSIONS $ npx wrangler d1 execute stub-db --file migrations/001_init.sql # ship it $ npx wrangler deploy # → https://stub.yourdomain.com (add route in dashboard)

11 what could go wrong

Abuse → short-link spam

Mitigated by: owner-only creation for links. Public endpoint is notes-only, Turnstile-gated, rate-limited via KV.

Outbound email deliverability

Magic links go to ONE address you control. If Email Workers have trouble, fallback is a tiny Resend (3k/mo free) binding.

D1 beta / data loss

D1 is GA but still maturing. Nightly cron exports a JSONL dump to R2 (R2 has 10 GB free) for disaster recovery.

Domain / cost creep

Only recurring cost is your domain (~$10/yr). Everything else stays on free tier unless usage explodes — in which case, good problem.
